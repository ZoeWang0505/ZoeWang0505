"Le projet est un projet de développement d'une application mobile par l'organisme MONA, visant la découverte de l'art public, mené par le Département d'histoire de l'art et d'études cinématographiques, en coopération avec le Département d'informatique et de recherche opérationnelle . Durant la session, je suis recruté comme développeur iOS. Pendant mon mandat, je travaillerais en utilisant le framework SwiftUI et le langage Swift avec XCode développé par Apple.

Mandats du Projet

Le mandat pour le projet consiste à publier la nouvelle version d' IOS de MONA pour offrire des nouvelle features:

  • se connecter /déconnecter

  • register pour un nouveau compte

  • mis-a-jour par l' ancien utilisateur et ses données.

Mettre à jour des données systématiquement dans application mobile en utilisant les nouvelles APIs lastUpdatedArtworks et lastUpdatedPlaces, lastUpdatedHeritages.

Le but est d'améliorer d'expérience d’utilisateurs par:

  • Retrouver des collections si utilisateurs se connecter par autre équipement mobile.

  • Réduire le temps de téléchargement et des taille de données qui ont besoin de mis-a-jour par request à serveur.

  • Supporter tous les équipements d' IOS qui ont varié de tailles d' écran, version supportée à partir de 14 IOS.

Introduire de nouveaux APIs

J'ai aussi pris la tâche d' introduire de nouveaux APIs dans IOS: lastUpdatedArtworks et lastUpdatedPlaces, lastUpdatedHeritages. L’idée est que les données du système vont être de plus en plus grandes, chaque fois télécharger des données n'est pas nécessaire, mais si il y a des modifications ou nouvelles données, il faut toujours mettre à jour dans l'application.

Pour le faire, j’ai besoin de comprendre des situations actuel:

  1. De quelle manière des données sont gardées dans l’application. Fiche de texte ou base de données locale.

  2. De quelle façon des lastUpdated APIs offrent des mises à jour. Ce sera seulement des propriétés changées ou chaque raccord avec un information complète, même si partie de propriétés sont changées.

  3. La date pour demander des données doit être gardée dans l’application localement pour IOS, il faut être similaire comme variable système pour Windows.

  4. J’ai aussi questionné la forme de la date, généralement il faut toujours dans la forme de UTC car c’est un standard pour résoudre des différences de fuseau horaire. Mais selon des documents de API, la format de date est yyyy-mm-dd n’est pas précisé suffisamment.

Après avoir rechercher dans des métiers, je conclus comme suit:

  1. Les données sont dans format de JSON et gardées localement comme documents.
  2. APIs offrent des données avec un information complète, c'est-à-dire, il n’a pas necessaire de comparer chaque raccord propriété à propriété, la solution est simplement lire le document de JSON, trouver le ancien raccorder par id, le replacer avec le nouveau.
  3. Swift offre NSDate pour un UTC date et UserDefault pour garder des variables système.
  4. J’ai demandé à Jianxin qui travaille pour serveur de vérifier si des APIs utilisant UTC aussi.

Implémenter les fonctions n'est pas compliqué après recherche. Mais il est encore deux parties de logique à modifier après tester:

  1. J’ai utilisé un variable pour garder le valeur de date la plus récente.Mais si un des trois APIs n’a pas répondu succès, il va avoir du problème pour la prochaine. Donc j’ai changé le code pour chaque API avec une variable différente.
  2. N’importe la première fois se connecter ou mise à jour du donné, si le demand est répondu success, le nouvelle date pour porchian a besoin de mise à jour, si non, gardé l' ancienne valeur.

 

Tester et bug fixer pour nouvelle UI:

La teste est longue. J’ai pris une semaine pour tester, noter tous les problèmes que j’ai trouvés, une autre semaine pour les fixer, et recommencer encore. J’ai testé pour fonctionnalités comme suit:

  • Page de bienvenue

C’est pour l' application n’est pas encore se connecter un compte, et aussi après le nouvelle installation.

  • Page de mise à jour

Cette page est pour les utilisateurs de l'ancienne version. L’ancienne version de IOS ne peut pas avoir un compte dans serveur, mais garder des collections localement. Il faut avoir leur données soumettre à serveur après qu' ils ont créé un compte.

  • Page de register

Similaire à page de mise à jour, mais il est pour les nouveaux utilisateurs.  

  • Page de télécharger du donné

Il faut prendre du temps pour télécharger du donnés personnel et des données systématiquement, après se connecter successivement.

Si il y a des erreurs à télécharger, l'utilisateur peut refaire une fois.

  • Function de se deconnecter

Cette fonctionnalité est dans page de paramètre. Quand l’utilisateur décide de se déconnecter, il faut enlever ses données personnelles et puis afficher le Page de bienvenue.

Comme une développeuse, au début j’ai pensé tester c’est facile. J’ai focuse sur de logique, tester tous les cas pour toutes les situations. Quand j’ai mis l’application sur Testflight pour testeurs interne pour la première fois, j’ai cru que c'était parfait. Mais ensuite, j'ai reçu beaucoup de rapports pour les bogues de l'interface utilisateur, et Lena m'a recommandé de vérifier soigneusement avec le document Figma. Je l'ai fait. J’ai trouvé que c’est vraiment différent entre de l'implémentation de UI et le document désigne: le font, les taille d' image, les style de boutons, les textes sur les pages et des boutons… Franchement, comme a developeur depuis longtemps, je n'ai jamais regardé attentivement une interface utilisateur comme celle-là. Et Prof. Guy Lapalme m’aide tester aussi sur le plus petit écran, iphone 6.

J’ai fait le bilan pour tous les problèmes que j’ai fixé:

  1. J’ai défini des constantes pour des tailles, des couleurs, des rayons de coin, et utiliser pour tous les composants dans le même style. Donc en changeant seulement les constantes, nous pouvons avoir tous les composants changeants.
  2. Mettre des positions et des tailles en parentage de l'écran d’appareil. Donc nous avons des valeurs calculées dynamiquement pour tous les appareils différents.
  3. Les désignations de réponse de API a parfois le même code d’erreur mais pour des raisons différentes, en écrit en français. Le solution orignal dans code est trouvé les même phase en englais correspondant à la phase en francais. C’est une risque, si il y a un petit changement dans serveur, il vas causer des bugs pour ne peut pas afficher des messages valide. J’ai utilisé des code d'erreurs seulement et j’ai défini des messages pour des réunions d' erreur. Par exemple, si code 422 retourne si le nom d'utilisateur n'existe pas, ou mauvais mot de passe, une message de <nom d'utilisateur n'existe pas, ou mot de passe est incorrect> vas afficher, à place d’un message general <erreur du serveur>.
  4. Parfois j’arrive avec des données par serveur mises à jour incorrectement, ces problèmes cause des erreurs dans mobiles. J’ai aidé Jianxin pour trouver les problèmes, et par cela, j'ai mieux pour des structures de données de MONA.

Une problème noté pour IOS token:

Nous avons vu le problème quand nous avons enlevé l’app de MONA sans se déconnecter, puis le télécharger et ouvourir, nous ne pouvons pas l’utiliser car l’app a nous considéré se connecté, mais rien de données existe.  Après avoir recherché sur internet, je sais que c’est à cause du token d’utilisateur est gardé dans cache locale de système, nous avons de logique pour le nettoyer quand l’utilisateur se déconnecte d’application, mais pas de chance pour le nettoyer quand l’utilisateur enlève l'application. Et la token ancien est encore valide jusqu'à ce que l'utilisateur se connecte avec un autre appareil, selon Apple developer. La solution est de garder des tokens dans serveur a la place de cache locale, et serveur vérifier le token pour chaque fois se connecter des données et générer un nouveaux token dans serveur pour prochaine fois.  mais c’est un autre level de solution sécurité nous pouvons l’avoir dans le future.
