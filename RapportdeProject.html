« Le projet est un projet de développement d’une application mobile par l’organisme MONA, visant la découverte de l’art public, mené par le département d’histoire de l’art et d’études cinématographiques, en coopération avec le département d’informatique et de recherche opérationnelle. Durant la séance, je suis recruté comme développeur iOS. Pendant mon mandat, je travaillerais en utilisant le cadre d’applications SwiftUI et le langage Swift avec XCode développé par Apple.

Mandats du Projet

Le mandat pour le projet consiste à publier la nouvelle version d’ IOS de MONA pour offrir de nouvelles fonctionnalités:

  • Se connecter ou se déconnecter

  • Enregistrer pour un nouveau compte de MONA

  • Mis à jour par l’ ancien utilisateur et des données.

Mettre à jour des données systématiquement dans application mobile en utilisant les nouvelles APIs lastUpdatedArtworks et lastUpdatedPlaces, lastUpdatedHeritages.

Le but est d’améliorer d’expérience d’utilisateurs par:

  • Retrouver des collections quand l’utilisateur se connecte par autre équipement mobile.

  • Réduire le temps de téléchargement et de taille du donné, quand mis à jour par demande à serveur.

  • Prends en charge tous les types d’appareils IOS qui ont varié de tailles d’ écran, avec les versions prises en charge à partir de 14 IOS.

Introduire de nouvelles APIs

J’ai aussi pris le travail d’ introduire de nouvelles APIs dans IOS: lastUpdatedArtworks , lastUpdatedPlaces, lastUpdatedHeritages. L’idée est que les données du système vont être de plus en plus grandes, chaque fois télécharger des données n’est pas nécessaire, mais s’il y a des modifications ou nouvelles données, il est toujours nécessaire de mise à jour dans l’application.

Pour le faire, j’ai besoin de comprendre des situations réalité:

  1. De quelle manière des données sauvegardées dans l’application ? Fiche de texte ou base de données locale?

  2. De quelle façon des lastUpdated APIs pour des mises à jour ? Ce sera seulement des propriétés changées ou chaque donne avec une information complète, même si une partie de propriétés sont changées.

  3. Le format de la date sauvegardée dans l’application localement pour IOS.

  4. J’ai aussi contesté la forme de la date, généralement c’est mieux dans la forme d’UTC, car c’est un standard pour résoudre des différences de fuseau horaire. Mais selon des documents d’API, le format de date est yyyy-mm-dd, n’est pas précisé suffisamment.

Après avoir recherché dans des métiers, je conclus comme suit:

  1. Les données sont dans un format de JSON et sont gardées localement comme documents.
  2. Les APIs offrent des données avec une information complète, c’est-à-dire qu’il n'est pas nécessaire de comparer des données de propriété à propriété, la solution est qu’on peut simplement lire le document de JSON, trouver l’ancien raccorder par id, le replacer avec le nouveau.
  3. Swift offre NSDate pour une UTC  et UserDefault pour garder des variables système.
  4. J'ai demandé à Jianxin de vérifier si les API utilisent le même format.

Implémenter les fonctions n’est pas compliqué après recherche. Mais il est encore deux parties de logique à modifier après test:

  1. J’ai utilisé une variable pour sauvegarder la valeur de date la plus récente.Mais si une des trois APIs n’est pas répondue avec succès, on va avoir du problème pour la prochaine fois . Donc j’ai changé le code pour chaque API avec une variable différente.
  2. Si c'est la première fois que vous vous connectez ou que vous mettez à jour le compte, après la réponse de demande succès,la nouvelle date enregistrée pour la prochaine utilisation, sinon, gardé l’ ancienne valeur.

 

Tester et fixer pour nouvelle UI:

Le test est long. J’ai pris une semaine pour tester, noter tous les problèmes que j’ai trouvés, une autre semaine pour les fixer, et recommencer. J’ai testé pour fonctionnalités comme suit:

  • Page de bienvenue

C’est pour des utilisateurs qui ne sont pas connectés à un compte, et aussi après la nouvelle installation.

  • Page de mise à jour

Cette page est pour les utilisateurs de l’ancienne version. Par l’ancienne version d’IOS, on ne peut pas créer un compte dans serveur, mais garder des collections localement. Il faut avoir leurs données soumises au serveur après qu’ ils ont créé un compte.

  • Page de registre

C'était similaire avec la page de mise à jour, mais il est pour les nouveaux utilisateurs.  

  • Page télécharger du donné

Il faut prendre du temps pour télécharger des données personnelles et des données systématiquement, après se connecter successivement.

S’il y a des erreurs à télécharger, l’utilisateur peut refaire une fois.

  • Fonction de se déconnecter

Cette fonctionnalité est implémentée dans la page de paramètre. Quand l’utilisateur décide de se déconnecter, il faut enlever ses données personnelles et puis afficher la page de bienvenue.

Comme une développeuse, au début j’ai pensé que tester c’est facile. J’ai concentré sur logique, et j’ai testé tous les cas pour toutes les situations. Quand j’ai mis l’application sur Testflight pour testeurs internes pour la première fois, j’ai cru que c’était parfait. Mais ensuite, j’ai reçu beaucoup de rapports pour les bogues de l’interface utilisateur, et Lena m’a recommandé de vérifier soigneusement avec le document Figma. Je l'ai suivi et puis j’ai trouvé que c’est vraiment différent entre de l’implémentation de UI et le document désigne: la police de caractère, les tailles d’ image, les styles de boutons, les textes sur les pages et des boutons… Franchement, comme une développeuse depuis longtemps, je n’ai jamais regardé attentivement une interface utilisateur comme celle-là. Et Prof. Guy Lapalme m’aide tester aussi sur le plus petit écran, iPhone 6.

J’ai fait le bilan pour tous les problèmes que j’ai fixé:

  1. J’ai défini des constantes pour des tailles, des couleurs, des rayons de coin, et utiliser pour tous les composants dans le même style. Donc en changeant seulement les constantes, nous pouvons avoir tous les composants changeants.
  2. Mettre des positions et des tailles en parentage de l’écran d’appareil. Donc nous avons des valeurs calculées dynamiquement pour tous les appareils différents.
  3. Les désignations de réponse d’API ont parfois le même code d’erreur, mais pour des raisons différentes, en écrivent en français. La solution originale qui cherche les mêmes phases en anglais correspondant à la phase en français. Cela est un risque que, si l’on modifie les messages pour serveur, il aille causer des bogues pour ne pas afficher des messages valides. J’ai utilisé des codes d’erreurs seulement et j’ai défini des messages pour une réunion d' erreurs. Par exemple,avec le code 422 pour que le nom d’utilisateur n’existe pas, ou que mauvais mot de passe, un message de <le nom d'utilisateur n'existe pas ou le mot de passe n'est pas valide> vas afficher, à place d’un message général <erreur du serveur>.
  4. Parfois, j'ai rencontré le problème de la mise à jour incorrecte des données sur le serveur. Ce qui m'a donné une erreur inattendue dans les mobiles. J’ai aidé Jianxin pour trouver les problèmes, et par cela, j’ai mieux pour des structures de données de MONA.

Donc pour tester cet API, j’ai implémenté la page de mot de passe oublié, donc nous pouvons publier une solution complète.

Un problème noté pour IOS token:

Nous avons vu le problème quand nous avons enlevé l’application de MONA sans nous déconnecter, puis le télécharger et ouvrir, nous ne pouvons pas l’utiliser, parce que l’application considère qu’il s’est connecté, mais rien de donné n’existe.  Après avoir cherché sur internet, je sais que c’est à cause du token d’utilisateur sauvegardé dans le cache local, nous avons la logique pour le nettoyer quand l’utilisateur se déconnecte en application, mais pas de chance pour le nettoyer quand l’utilisateur enlève l’application. Et le token ancien est encore valide jusqu’à ce que l’utilisateur se connecte avec un autre appareil, selon Apple développer. La solution est de sauvegarder des tokens à serveur,  à la place du cache local, et plus avec le serveur vérifie le token à accéder aux données, et génère un nouveau token pour la sécurité.  Mais c’est un autre niveau de solution sécurité que nous pouvons l’avoir dans le futur.
